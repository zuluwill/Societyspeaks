## Major architectural change

### What changed

Original design:
- Showed TrendingTopics (clustered news stories)
- Displayed coverage analysis per topic
- Lazy-loaded perspectives for topics

Current design:
- Shows individual NewsArticle items directly
- Three-column layout (left/center/right)
- Source filtering and content type filtering (news vs podcasts)
- No topic clustering or coverage analysis per story

## Critical issues

### 1. No pagination or limits
Location: `app/news/routes.py:101-108`

```python
# Fetch all articles from last 24 hours - no hard limit
articles = NewsArticle.query.filter(
    NewsArticle.published_at >= cutoff,
    NewsArticle.source_id.in_(source_ids)
).order_by(NewsArticle.published_at.desc()).all()
```

Problem:
- Could load thousands of articles
- High memory usage
- Slow page load
- Large JSON cache payload

Impact: Will not scale with volume.

Recommendation: Add pagination or a hard limit (e.g., top 500 by relevance_score).

---

### 2. Cache serialization overhead
Location: `app/news/routes.py:186-189`

```python
cache.set(DASHBOARD_CACHE_KEY, json.dumps(result), timeout=DASHBOARD_CACHE_TTL)
```

Problem:
- Serializing/deserializing large article lists is expensive
- JSON parsing on every request adds latency

Recommendation: Consider caching IDs only and fetching articles on demand, or use a more efficient format.

---

### 3. Missing index on published_at
Location: `app/news/routes.py:103-108`

```python
articles = NewsArticle.query.filter(
    NewsArticle.published_at >= cutoff,
    NewsArticle.source_id.in_(source_ids)
).order_by(NewsArticle.published_at.desc()).all()
```

Problem:
- Without an index on `published_at`, this query will be slow with many articles

Recommendation: Ensure a composite index on `(published_at DESC, source_id)` exists.

---

### 4. Inconsistent data model
Location: Multiple files

Problem:
- Dashboard shows `NewsArticle` items
- API endpoint `/api/news/perspectives/<topic_id>` still expects `TrendingTopic`
- Two different data models for the same feature

Impact: Confusing architecture; the perspective API may not align with the new article-based UI.

---

## Recommendations

### Immediate (before production)

1. Add pagination or a hard limit (top 500 articles)
2. Verify database indexes on `published_at` and `source_id`
3. Resolve the article vs topic model mismatch
4. Test with realistic data volumes (1000+ articles)

### Short term (week 1)

1. Optimize cache format (cache IDs, not full objects)
2. Add database flag for podcasts instead of keyword matching
3. Consider reintroducing topic clustering or story grouping
4. Add metrics for cache hit rate and query performance

### Long term

1. Decide: articles-only or hybrid (articles + topics)?
2. If keeping articles-only, remove unused topic-related code
3. Consider adding story clustering back as an optional view
4. Add analytics for filter usage
