Step 3: Create or lookup a discussion

Store the discussion ID in your CMS for consistent embeds. Use an Idempotency-Key on create to safely retry failed requests. You can create discussions with either article_url or external_id (for non-article contexts such as campaign hubs or live blogs).

Prefer Python or Node.js over raw HTTP? Download our SDKs → 

curl "{{ base_url }}/api/discussions/by-article-url?url=https://example.com/article" \
  -H "X-API-Key: sspk_test_..."

curl -X POST "{{ base_url }}/api/partner/discussions" \
  -H "X-API-Key: sspk_test_..." \
  -H "Idempotency-Key: unique-request-id-123" \
  -H "Content-Type: application/json" \
  -d '{"article_url":"https://example.com/article","title":"Example title","excerpt":"..."}'

curl -X POST "{{ base_url }}/api/partner/discussions" \
  -H "X-API-Key: sspk_test_..." \
  -H "Idempotency-Key: unique-request-id-124" \
  -H "Content-Type: application/json" \
  -d '{"external_id":"observer-cms-12345","title":"Campaign discussion","seed_statements":[{"content":"This is a valid seed statement for this topic.","position":"neutral"}]}'

Step 6: Configure webhooks and analytics export

Set up webhook callbacks in the dashboard to avoid polling and keep Observer systems in sync. Verify X-SocietySpeaks-Signature on your receiver and treat X-SocietySpeaks-Event-Id as idempotent. 

curl "{{ base_url }}/api/partner/analytics/usage-export?days=30&env=all&format=json&page=1&per_page=100" \
  -H "X-API-Key: sspk_live_..."

Step 7: Set embed submission policy per discussion

Start with vote/report-only (default). For selected discussions, owners/admins can enable inline reader statement submissions.

  • Go to Dashboard -> Discussions -> open a discussion -> Embed submission policy.
  • Keep disabled for most article pages; enable for campaign/live-blog workflows.
  • When enabled, readers can submit 10-500 character statements directly in embed.
